# Compile-time ECS с поддержкой нескольких миров и сериализацией

Изначально это был прототип ECS из моего движка github.com/Pepengineers/PEPEngine2 , сделанный весной, я его доработал и сделал сериализацию.

---

## Что реализовано

- Entity - обёртка uint32_t;
- базовый маркер компонента - Component;
  - TransformComponent;
  - VelocityComponent;
  - HealthComponent;
  - NameComponent;
- World - один мир/уровень, обёртка над ECS-хранилищем;
- создание и удаление entity;
- добавление, получение, проверка и удаление компонентов;
- Защита ECS от некорректного доступа;
- System - базовый класс для систем ECS;
  - MovementSystem, двигающая сущности с TransformComponent и VelocityComponent;
  - PrintTransformSystem, выводит состояние transform-компонентов;
- SceneManager хранит миры и тикает системы по их приоритету, а не последовательно по мирам;
- загрузка сцены из .yaml файлов;
- сохранение сцены обратно в .yaml;
- обработка ошибок при загрузке сцены;
- демо приложение в App.cpp.

---

## Структура проекта

- App — демо-приложение, зависит от ECS.

- ECS — сама ECS, зависит от ThirdParty/RapidYAML.
  - ECS/Components — компоненты.
  - ECS/Systems — системы.

- ThirdParty — проект для сторонних зависимостей.
  - ThirdParty/RapidYAML — парсер .yaml файлов.

- Assets — общая папка ресурсов.
  - Assets/Scenes — сцены.

---

Я сделал compile-time переключатель между режимами хранения самих значений в пулах и указателей на значения.
По умолчанию используется value storage, так как он лучше подходит для ECS по локальности кеша. Pointer storage оставлен так как этого требует тз. API к ECS не зависит от режима.

В pointer storage для хранения компонентов используется std::unique_ptr<T>, а не std::shared_ptr<T>, потому что компонент не должен иметь нескольких владельцев. Он принадлежит конкретному ComponentPool<T>, а его время жизни зависит от его entity. Когда entity удаляется, соответствующий компонент тоже удаляется из пула.
Для повторного AddComponent используется ComponentStorageTraits::Assign(), в котором в pointer storage режиме обновляется уже существующий объект без создания нового указателя и без лишней аллокации памяти.

Для сериализации выбрана yaml разметка из-за большой вложенности данных в ECS. У yaml лучше читаемость и вес файла меньше.
Обработка ошибок есть на всём пути загрузки сцены, приложение не ложится при плохих файлах сцены.

После клонирования git submodule update --init --recursive.

Проект собирается в Visual Studio / Rider с установленными Visual C++ Build Tools.

Поддерживаемые конфигурации:

- Debug | x64
- Release | x64

Запускаемый проект - App.

Демо-сцена находится по пути:
Assets/Scenes/DemoScene.yaml

Демо-приложение:
- Создаётся SceneManager (владелец сцены, состоящей из миров);
- Загружается сцена из Assets/Scenes/DemoScene.yaml;
- В консоль выводятся восстановленные миры, entity и компоненты;
- Выполняется 3 тика;
- MovementSystem изменяет позицию нужных entity;
- PrintTransformSystem выводит transform-компоненты;
- После тиков сцена сохраняется обратно в YAML (путь относительно запускаемого App.exe, Assets\Scenes\SavedDemoScene.yaml);
